Layer 01

Browser-Side Tracking via GTM Web Container

The GTM Web Container sits on the storefront and captures user interactions in real time. Each tag is configured with precise trigger conditions, data layer variables, and event parameters mapped to Meta's standard event schema.

PageView ViewContent AddToCart InitiateCheckout Purchase
⚠️ This browser pixel — no matter how perfectly configured — is vulnerable. It can be blocked, throttled, or rendered useless by a single privacy toggle on a user's iPhone. Layer 1 is necessary but never sufficient.
Layer 02

Server-Side Tracking via GTM Server Container + Meta CAPI

This is where the architecture becomes enterprise-grade. A dedicated GTM Server-Side Container is deployed on a cloud environment, acting as a secure, first-party intermediary.

1

A user completes an action on ZenithCart (e.g., a Purchase)

2

The GTM Web Container fires the event and simultaneously sends the event payload to the Server Container via a first-party endpoint

3

The Server Container receives raw event data — enriched with hashed email, hashed phone, IP address, user agent, fbc, fbp cookies

4

The Server Container sends data directly to Meta via the Conversion API — server-to-server. No browser involved. No JavaScript to block. No cookie to truncate.

🛡️ Bypass Ad Blockers

The CAPI call is a server-to-server HTTPS request. Invisible to browser extensions, ITP, and ATT. Cannot be blocked.

📱 Survive iOS 14+ Opt-Outs

Even when users opt out at device level, server-side events enriched with hashed first-party data still reach Meta.

🔒 First-Party Data Ownership

Cookies are classified as first-party by all major browsers. Extends cookie lifetimes from 7 days back to full duration.

Layer 03

Intelligent Event Deduplication

Sending the same event from both browser pixel AND server creates a dangerous risk: double-counting. Inflated conversions corrupt optimization signals in the opposite direction.

①

A unique event_id is dynamically generated using a Custom Variable Template in GTM at the exact moment of user interaction

②

This identical event_id is attached to both the browser pixel tag and the Server Container payload simultaneously

③

Meta recognizes the matching event_id and counts the conversion exactly once — while benefiting from the richer server-side data

⚡ This is not optional. This is mandatory for any legitimate CAPI implementation. And it is fully operational inside ZenithCart.
